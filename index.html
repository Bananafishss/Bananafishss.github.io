重载的目的

对于一个自定义的类变量（对象）来说，无法直接使用操作符运算，比如对于自定义的 Complexe 类，
‘’’cpp
class Complexe{
private:
double x;
double y;
};
‘’’
对于Complexe a,b; 无法：a+b；所以需要重新定义操作符。

操作符重载的方法

  1. 第一种重载是把操作符重载看作是classe的成员函数（方法）：
    ‘’’cpp
    //A Op B 写作 A.operatorOp(B)
    t1 == t2 // 等价于: t1.operator==(t2)
    t1 += t2 // 等价于: t1.operator+=(t2)
    ‘’’
  2. 第二种重载是把操作符重载当成一个类外的普通函数，然后将被运算的数据作为参数传进操作符函数里：
    ‘’’cpp
    //A Op B 写作 operatorOp(A,B)
    t1 + t2 // 等价于 : operator+(t1,t2)
    t1 * t2 // 等价于 : operator*(t1,t2)
    ‘’’

  1. I2C协议
    I²C 有两根信号线：
    SCL：时钟线（Clock），一般由主机（Master）输出
    SDA：数据线（Data），双向传数据
    GND 共地
    下面展示一个通过I2C协议读传感器数据的例子：

    ‘’’c

#include <Wire.h>//引入 I2C 库
const uint8_t MPL3115A2_ADDR = 0x60;//传感器 I2C 端口地址
bool i2cWrite8(uint8_t addr, uint8_t reg, uint8_t val) {
Wire.beginTransmission(addr);//调用wire库里的开始函数来知道要操作的是哪个外设，这里的addr就是传感器地址
Wire.write(reg);//具体到传感器外设的地址为reg的寄存器
Wire.write(val);//往reg地址的寄存器里写val
return (Wire.endTransmission() == 0);//结束
}//这是一个修改外设指定的寄存器值的函数

bool i2cReadBytes(uint8_t addr, uint8_t reg, uint8_t* buf, uint8_t len) {
Wire.beginTransmission(addr);
Wire.write(reg);
if (Wire.endTransmission(false) != 0) return false;//I2C里的repeated start

uint8_t got = Wire.requestFrom(addr, len);//从刚刚的寄存器里读len个字节。
if (got != len) return false;

for (uint8_t i = 0; i < len; i++) buf[i] = Wire.read();//把刚才从寄存器收到的每一个字节，一个一个读出来，存进我给的数组 buf[] 里。
return true;
}//从外设连续读取 len 个字节的数据，并存进给它的缓冲区 buf 里

bool mpl3115a2Init() {
// PT_DATA_CFG = 0x07 (enable event flags)打开数据就绪 / 覆盖 这些事件标志功能
if (!i2cWrite8(MPL3115A2_ADDR, 0x13, 0x07)) return false;

// CTRL_REG1 = 0xB9 (ALT=0 temp mode, OSR, active)进入温度模式 + 128x 过采样 + 开始工作
if (!i2cWrite8(MPL3115A2_ADDR, 0x26, 0xB9)) return false;

delay(10);
return true;
}//初始化

bool readTemperatureC(float &tempC) {
uint8_t buf[2];
if (!i2cReadBytes(MPL3115A2_ADDR, 0x04, buf, 2)) return false;
int8_t msb = (int8_t)buf[0];
uint8_t lsb = buf[1];
float frac = (float)(lsb >> 4) / 16.0f; //把 lsb 右移 4 位，只保留“高 4 位”。
tempC = (float)msb + ((msb >= 0) ? frac : -frac);//把整数和小数部分拼接起来。
return true;
}

  1. SPI 接口
    SPI 即串行外围设备接口，是一种适用于短距离、设备到设备的高速全双工同步通信接口，通信结构为一主多从，硬件接口上通常由4根信号线组成，简单易用；
    通信模式：
    标准SPI接口一般使用4条线(4个信号)：

NSS/CS：主机片选从机使能信号，通常低电平有效由主机控制；

SCLK：时钟信号，主机产生用于控制数据交换的时机以及速率；

MISO：主机数据输入，从机数据输出，主要用于SPI 设备接收数据；

MOSI：主机数据输出，从机数据输入，主要用于 SPI 设备发送数据。

具体流程：
① 主机拉低目标通信从机片选SS信号线，表示与该设备进行通信；

② 主机产生SCLK时钟信号，告诉从机要交换数据；

③ 主机(Master)把需要发送的数据写入到发送数据缓存区(Menory)后，再打入串行移位寄存器(一般为8位)，串行移位寄存器通过MOSI信号线将字节一位一位地移出传送到从机，同时，MISO接口接收到的数据经过串行移位寄存器一位一位的移到接收缓存区。

从机(Slave)也将其串行移位寄存器中的数据通过MISO信号线发送到主机，同时通过MOSI信号线接收主机发送的数据。

如此一来，两个移位寄存器中的数据就被交换。

  1. CAN 协议
    这是一个多主总线的协议，就想广播那样，一个发射端发出数据，所有的其他接收端都可以接收数据。
    按 标识符长度 划分数据帧，分为标准帧和扩展帧两类：
    标准帧（Standard Frame）：使用 11 位标识符（ID）（共 44～108 位，取决于数据长度）
    [SOF] [11位ID] [RTR] [IDE=0] [r0] [DLC(4位)] [Data(0～8字节)] [CRC] [ACK] [EOF]
    扩展帧（Extended Frame）：使用 29 位标识符（ID）
    [SOF] [11位基础ID] [SRR=1] [IDE=1] [18位扩展ID] [RTR] [r1][r0] [DLC(4位)] [Data(0～8字节)] [CRC] [ACK] [EOF]

  2. [SOF] — Start of Frame（帧起始）
    长度：1 位
    电平：显性（0）
    作用：标志一帧报文的开始，同步所有节点的位时序。
    注意：所有 CAN 节点检测到总线空闲后，若看到 SOF（下降沿），即开始接收该帧。

  3. [11位基础ID] — Base Identifier（基础标识符）
    长度：11 位
    作用：扩展帧 ID 的高 11 位，与标准帧 ID 格式兼容。
    意义：用于仲裁优先级的前 11 位比较；也用于与标准帧共存时的优先级判定。

  4. [SRR] — Substitute Remote Request（替代远程请求位）
    长度：1 位
    固定值：隐性（1）
    作用：
    在扩展帧中代替标准帧 RTR 的位置。
    确保当标准帧与扩展帧使用相同 11 位 ID 前缀时，标准帧优先级更高（因为标准帧 RTR 可为 0，而 SRR=1 是隐性，会被显性 0 覆盖而退出仲裁）。
    仅存在于扩展帧中。

  5. [IDE] — Identifier Extension Bit（标识符扩展位）
    长度：1 位
    值含义：
    0（显性） → 标准帧
    1（隐性） → 扩展帧
    作用：区分标准帧与扩展帧的关键标志位。
    位置：紧跟在 11 位基础 ID 之后。
    在扩展帧中，IDE = 1。

  6. [18位扩展ID] — Extended Identifier（扩展标识符）
    长度：18 位
    作用：扩展帧 ID 的低 18 位，与前面 11 位组合成 29 位完整 ID。
    总 ID 范围：0x00000000 ～ 0x1FFFFFFF（共 2²⁹ 个唯一标识符）
    用途：支持更多设备或更细粒度的消息分类（如汽车中不同 ECU 的子功能）。

  7. [RTR] — Remote Transmission Request（远程传输请求位）
    长度：1 位
    值含义：
    0（显性） → 数据帧（包含 Data 字段）
    1（隐性） → 远程帧（无 Data 字段，请求其他节点发送该 ID 的数据）
    注意：在扩展帧中，RTR 位于扩展 ID 之后。

  8. [r1][r0] — Reserved Bits（保留位）
    长度：2 位（r1 和 r0）
    当前值：必须发送为显性（0）（按 CAN 2.0 规范）
    作用：为未来协议扩展预留。
    接收方行为：若收到非 0 值，可能视为格式错误（取决于控制器实现）。
    🔧 在 CAN FD 中，这些位被重新定义（例如 r0 变为 BRS 位），但在经典 CAN 中必须为 0。

  9. [DLC] — Data Length Code（数据长度代码）
    长度：4 位
    取值范围：0 ～ 15（二进制 0000 ～ 1111）
    实际数据长度映射（经典 CAN）：
    表格
    DLC 值 实际数据字节数
    0～8 = DLC 值

9～15 仍为 8 字节（但某些 CAN FD 控制器可能用 DLC 表示更大长度）
作用：告诉接收方本帧包含多少字节的有效数据。
9. [Data(0～8字节)] — 数据字段
长度：0 到 8 字节（由 DLC 决定）
发送顺序：先发送高字节（MSB），每个字节内先发高位（bit7 → bit0）
内容：应用层要传输的实际信息（如传感器值、控制命令等）
注意：远程帧中此字段不存在。
10. [CRC] — Cyclic Redundancy Check（循环冗余校验）
组成：
15 位 CRC 校验码：基于从 SOF 到 Data 的所有位（不含填充位）计算得出
1 位 CRC 界定符（Delimiter）：固定为 隐性（1）
总长：16 位
作用：检测传输过程中是否发生位错误。接收方重新计算 CRC 并比对，不一致则报错。
11. [ACK] — Acknowledgement（应答段）
组成：
ACK Slot（应答槽）：1 位
发送方发 隐性（1）
任一正确接收的节点必须在此位发 显性（0） 以确认
ACK Delimiter（应答界定符）：1 位，固定为 隐性（1）
总长：2 位
作用：至少一个接收者确认成功接收。若 ACK Slot 仍为 1，发送方认为传输失败。
12. [EOF] — End of Frame（帧结束）
长度：7 位
电平：全部为 隐性（1）
作用：
明确标识本帧结束
提供帧间间隔（IFS）的起点
不允许位填充（即使出现 6 个连续 1 也不插入 0）
之后：进入 IFS（Interframe Space，3 位隐性），然后总线才可被新帧占用。

微处理器 PWM控制机械指针

指针角度和计数器计数之间的转换的两个问题：

  1. 角度超出180
  2. 系统上电时初始角度不一致

指针旋转角度和马达计数器之间的转换

“一个计数对应指针轴多少度”函数

‘’’c
static inline float degPerDriveCountOnShaft() {
return 360.0f / (float)ONE_REV_COUNTS / GEAR_RATIO;
}
‘’’
指针轴每个 count 转的角度 = 360 / 12 / 52.734

wrap180 函数是将角度限制到-180 180

角度 → 计数上面的逆函数

首先我们有 不含校准 offset
‘’’
static inline long shaftDegToDriveCount(float deg) {
return lroundf(deg / degPerDriveCountOnShaft());
}//count = 角度 / (每个 count 对应的角度)
‘’’
然后 改进版 角度 → 目标计数（含校准 offset:
‘’’c
long expectedDegToTargetCount(float expDeg) {
// expDeg == count*degPer + offset => count = (expDeg - offset)/degPer
float pure = wrap180(expDeg - calibOffsetDeg);
return lroundf(pure / degPerDriveCountOnShaft());
}//系统上电时，编码器的 0 点 ≠ 指针表盘的 0 度,所以引入了 calibOffsetDeg（校准角偏置）
‘’’

  1. 先expDeg - calibOffsetDeg：把“表盘角度”换到“编码器坐标系”

  2. 然后用wrap180：避免角度跨 ±180 产生跳变

  3. 最后使用没校准的函数 除以 degPerDriveCountOnShaft() 得到 count（整数）

驱动马达旋转到目标位置 也就是让count和目标计数targetCount之间的差值err为零

  1. 先读编码器的计数得到当前count
  2. 根据当前count 和目标count 之间的误差err 控制电机旋转 直到误差为0。

读取编码器“当前位置”的count

‘’’c
Encoder enc(PIN_ENC_A, PIN_ENC_B);
//这个函数就是当前电机（或驱动轮）的“角度位置”用计数表示。
long readCountDrive() {
long c = enc.read();//读出增量编码器累计计数（count）
if (INVERT_ENCODER) c = -c; //如果转向/计数方向反了，就翻转
return c;
}
‘’’

设置 目标计数targetCount

目标计数targetCount根据温度传感器的当前值得到，付给变量long targetCountDrive 。

误差计算

long err = targetCountDrive - c;
err > 0：说明还没转够，要往正方向转

err < 0：转过头了，要往反方向转

err = 0：刚好到位
最经典的闭环思路：控制误差趋近于 0

电机驱动

‘’’c
void setMotor(bool dirHigh, int pwm) {

}
‘’’
PIN_DIR：方向（HIGH/LOW）

PIN_EN：PWM（0~255，越大越用力）

3. 状态机：远的时候快跑，近的时候慢跑，最后轻推

1）粗调（coarse）：比例控制 u = Kp * err

‘’’

if (pwm > 0 && pwm < PWM_COARSE_MIN_FAST) pwm = PWM_COARSE_MIN_FAST; // 60

SLOW 模式：

if (pwm > 0 && pwm < PWM_COARSE_MIN_SLOW) pwm = PWM_COARSE_MIN_SLOW; // 55

最大 PWM 是为了别冲太猛、别过冲。

  3. 和指针的区别
    1. 定义引用：

    2. 引用的规则：

C指针以及结构体，内存空间

内存的类型

要理解指针的用途 首先要理解内存的类型，总的来说分为全局区，栈，堆。每种内存区域都对应专门的功能：

  1. 全局区，最常用的内存区域，全局变量，结构体等在此处储存，声明周期到程序结束为止。
  2. 栈，快存快销的区域，用于即时计算，最典型的就是函数内部声明的变量或者结构体，在函数结束后一同被销毁，无法传出函数外部。
  3. 堆，由coder自己决定它的生命周期，用malloc创建 用free释放。它与全局区创建的结构体的区别在于：全局区声明即确定它的内存大小而malloc创建的内存空间是在运行时才被决定的，，这为链表等方法的实现提供了可能。

用途及用法

1. 由于C语言函数不能改变函数外面的变量，所有在需要函数对全局的某个变量进行处理时，需要用指向该变量的指针作为参数（就是传入地址而不是变量的值）传入函数。例如：
'''void add1(int *x) {
    *x = *x + 1;
}'''
2. 一个函数只能return 一个值，在需要返回多个值时，需要在参数中加入传出时的指针，然后无需return，用指针指向需要被返回的值即可。例如：
''' void div_mod(int a, int b, int *quot, int *rem) {
    *quot = a / b;
    *rem  = a % b;
}'''
3. 传递结构体时，结构体内存占用很大，使用指针避免内存大量消耗。例如：'''void process(struct Data *d);'''
4. 传递数组时，数组本质上就是指针，所以外部数组可以在函数内部直接修改。例如：'''void process(struct Data *d);'''

结构体的初始化方法

  1. 用于在函数内部使用，不能把本结构体的地址传到函数外部。
  2. 对函数外面的结构体初始化，可以将元素挨个赋值或者将一个结构体变量赋给它。

Timer

一个微控制器中自动记数的硬件模块，它的结构依次由分频器 Prescaler，计数器 counter, 比较单元 Compare Unit 最后从比较单元输出一个 1.标志位Flag(TOV或OCF)给CPU 或 2.输出中断请求（Interrupt）给CPU 3.输出电平信号（PWM / 翻转 IO）给外部电路。

Timer的初始化

下面是Adruino MEGA 2560的Timer：
Timer0：Arduino 核心用来做 millis()/delay()，一般不用。

Timer1/3/4/5：16 位，一般用来做PWM/精确定时。

Timer2：8 位，也能用，但分辨率低一些。

用 Timer1 的 OC1A 引脚输出 PWM：
‘’’C
void timer1_pwm_init_20kHz_oc1a() {
pinMode(11, OUTPUT); // OC1A

// 1) 停止 Timer1
TCCR1A = 0;
/*Timer1 的“控制面板 A”
❌不输出 PWM
❌ OC1A / OC1B 引脚不被 Timer 控制
❌ 不启用任何比较输出
❌ WGM10 / WGM11（PWM 模式） = 0
/
TCCR1B = 0;
/❌ Timer1 不跑（CS=000）
❌ 没有时钟输入
❌ WGM12 / WGM13 = 0
*/
TCNT1 = 0;
//计数器从 0 开始数，不带“上一次残留的值”

// 2) Fast PWM, TOP = ICR1 (WGM13:0 = 14 => 1110)
TCCR1A |= (1 << WGM11);
TCCR1B |= (1 << WGM13) | (1 << WGM12);

// 3) 非反相输出：Clear OC1A on compare match, set at BOTTOM
TCCR1A |= (1 << COM1A1);

// 4) 设定频率：f_pwm = F_CPU / (N * (1 + TOP))
// 20kHz => TOP = F_CPU/(Nf) - 1
// 16MHz, N=1 => TOP = 16e6/(120e3) - 1 = 799
ICR1 = 799;

// 5) 初始占空比 0%
OCR1A = 0;

// 6) 启动：分频 N=1
TCCR1B |= (1 << CS10);
}

}

本实验主要保留白色线条，同时剔除黄色线条与反光区域

过滤逻辑：

白色：低饱和度 + 高亮度

黄色：中等饱和度 + 特定 Hue 范围，直接剔除

最终得到仅包含白色车道线的二值图像。

1.3 ROI 掩膜（Region of Interest）

去除天空、桌椅、道路外区域

使用五边形 ROI，上窄下宽，符合车辆视野形状

ROI 顶点根据图像比例动态计算

与二值图像做按位与 → 仅保留小车前方区域

好处：

大幅减少误检

降低计算量

提高左右车道的提取精度

1.4 车道区域筛选（Lane Region Filtering）

即便经过 ROI，仍可能有噪声或非车道区域

对每一行（y）统计横向亮度分布

自动估计车道的大致横向范围

动态带宽掩膜：

底部窄，上部宽

模拟透视变化，让算法能捕捉远处细小车道线

最终只保留主要车道区域，提高后续聚类成功率。

1.5 温和清理与线段拼接（Soft Cleaning & Merging）

(1) 温和清理

使用连通域分析

删除：

面积太小的噪点

横向过宽的块

保留：

合理纵向线条

(2) 拼接断裂线段

对左右两半图像进行形态学闭操作（closing）

补全车道线间的小断裂

结果为：干净、连续、可检测的左右两条车道线。

1.6 骨架化 + DBSCAN 聚类

处理流程：

Canny 边缘检测

闭操作

骨架化（Thinning） → 得到单像素宽度的主干线

DBSCAN 聚类 → 自动区分左右车道线

不需要指定簇数

噪声（label = -1）自动剔除

小簇（<50点）过滤

Silhouette Score 评估聚类质量

优点：

抗噪能力强

能适应车道线断裂、光照变化

自动找到左右两条主干线

1.7 曲线拟合（Curve Fitting）

对每个簇的点集进行二次多项式拟合：

x = a·y² + b·y + c

为什么不用 y = f(x)？

在弯道可能出现“一条 y 对应多个 x”的多值问题

而 y 始终单调递增，更适合建模 BEV 中的纵向车道结构

输出：

左车道拟合曲线

右车道拟合曲线

在 BEV 图像上叠加可视化

1.8 坐标生成、偏向角、中心偏移量（Lane Point Generation）

完成拟合后，需要将连续曲线离散化成像素点：

从 y = ymin → ymax 均匀采样

根据函数计算 x

得到结构化点集（按 y 递增排序）

额外计算：

两条车道线的方向角（朝向角）

左右车道线与中线之间的距离

小车相对中心线的偏移量

所有结果存储到 JSON，用于控制模块。

未来可通过真实赛道标定：

将 pixel → meter 转换为真实距离

2. 结果分析（Results）

基于 111 张模拟器图像的测试结果：

a) 时间性能

平均耗时：0.015 s/帧

标准差：0.00725 s

最快：0.009 s

➡ 支持高刷新率实时控制

b) 鲁棒性

c) 混淆情况

在仅看到一条车道线时仍可保持区域识别

左右簇未混淆比例：98.19%

聚类数量：

平均：2.1

标准差：0.3

最优：2

3. 结论（Conclusion）

整体算法在模拟器测试中表现稳定：

实时性强：支持高频控制循环

准确率高：93.69%

左右车道分离效果良好：98.19% 未混淆

能够处理车道丢失、部分遮挡等情况

适用于真实环境和模拟环境的车道检测任务。

4. 展望（Perspectives）

GitLab使用教程

GitLab使用教程

在将DonkeyCar项目推进到小组进行整合时，要用到学校的GitLab，所以今天就来顺便学习一下Git的基本用法。

基本流程

一、配置SSH 链接GitLab

  1. 首先使用学校邮箱创建个人账户 我的邮箱是lixi.wang@imt-atlantique.net 用户名是l24wang 这步很简单就暂且不提。

  2. 第二步，配置SSH key 这步对于每个账号只需要配置一次： ssh-keygen -t ed25519 -C "你的邮箱"
    ✅ -t ed25519 指定加密算法
    ✅ -C 是备注信息（写邮箱）
    ⛔ 不需要设置密码

  3. 对于Windows：启动 ssh-agent 并载入 key：

    Start-Service ssh-agent
    ssh-add $env:USERPROFILE\.ssh\id_ed25519

  4. 将key添加到GitLab: 复制在终端中生成的key，然后在网站的key选项那里手动添加。

  5. 测试key连接是否成功 ssh -T git@gitlab.imt-atlantique.fr
    成功显示： Welcome to GitLab, @l24wang!

二、克隆 GitLab 仓库到本地

  1. 先 cd 到想要项目到本地的目的文件夹里
  2. 用git clone git@gitlab.imt-atlantique.fr:仓库路径.git
    我这里是 git clone git@gitlab.imt-atlantique.fr:z23mahmo/2025-2026-ia-racing.git
    git@…是使用 SSH 方式下载的意思

三、把自己的文件加入仓库（本地）

复制粘贴你的文件到仓库文件夹里，注意现在你的文件还没有更新到线上仓库和别人共享，只能在本地查看。

四、把自己的文件更新到线上仓库

git add .
git commit -m "Add my code"
git push origin main
