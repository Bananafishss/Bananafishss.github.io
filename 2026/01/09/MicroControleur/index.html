微处理器 架构和接口协议

微控制器基本架构：

一个完整的工程系统包括：传感器 → 信号处理 → 控制 / 显示 → 执行器，MCU负责处理来自传感器和信号处理等外设（ADC模数转换器）的信号后发出指令给输出端口的外设。

MCU端口结构(外设)

GPIO 最简单的端口

GPIO是一组16个可以自由配置每一位电平的端口，有输入输出两种模式，配置代码如下：
DDRx ：用来配置一个端口的输入/输出方向。举例：DDRA = 0xFF //将A组GPIO每一个端口全部设置为 1 也就是输出模式。DDRB = 0x00//B组设为 0 则是输入模式
接下来初始化A组的输出值和B组的输入值为0：
PORTA = 0x00
PINB = 0x00

Port Serie 有通信协议的端口

不能自由设置每个端口的值，一般有tx和rx 它们的值电平由串口硬件自动控制。常用的端口类型有SPI / I2C / CAN

  1. I2C协议
    I²C 有两根信号线：
    SCL：时钟线（Clock），一般由主机（Master）输出
    SDA：数据线（Data），双向传数据
    GND 共地
    下面展示一个通过I2C协议读传感器数据的例子：

    ‘’’c

#include <Wire.h>//引入 I2C 库
const uint8_t MPL3115A2_ADDR = 0x60;//传感器 I2C 端口地址
bool i2cWrite8(uint8_t addr, uint8_t reg, uint8_t val) {
Wire.beginTransmission(addr);//调用wire库里的开始函数来知道要操作的是哪个外设，这里的addr就是传感器地址
Wire.write(reg);//具体到传感器外设的地址为reg的寄存器
Wire.write(val);//往reg地址的寄存器里写val
return (Wire.endTransmission() == 0);//结束
}//这是一个修改外设指定的寄存器值的函数

bool i2cReadBytes(uint8_t addr, uint8_t reg, uint8_t* buf, uint8_t len) {
Wire.beginTransmission(addr);
Wire.write(reg);
if (Wire.endTransmission(false) != 0) return false;//I2C里的repeated start

uint8_t got = Wire.requestFrom(addr, len);//从刚刚的寄存器里读len个字节。
if (got != len) return false;

for (uint8_t i = 0; i < len; i++) buf[i] = Wire.read();//把刚才从寄存器收到的每一个字节，一个一个读出来，存进我给的数组 buf[] 里。
return true;
}//从外设连续读取 len 个字节的数据，并存进给它的缓冲区 buf 里

bool mpl3115a2Init() {
// PT_DATA_CFG = 0x07 (enable event flags)打开数据就绪 / 覆盖 这些事件标志功能
if (!i2cWrite8(MPL3115A2_ADDR, 0x13, 0x07)) return false;

// CTRL_REG1 = 0xB9 (ALT=0 temp mode, OSR, active)进入温度模式 + 128x 过采样 + 开始工作
if (!i2cWrite8(MPL3115A2_ADDR, 0x26, 0xB9)) return false;

delay(10);
return true;
}//初始化

bool readTemperatureC(float &tempC) {
uint8_t buf[2];
if (!i2cReadBytes(MPL3115A2_ADDR, 0x04, buf, 2)) return false;
int8_t msb = (int8_t)buf[0];
uint8_t lsb = buf[1];
float frac = (float)(lsb >> 4) / 16.0f; //把 lsb 右移 4 位，只保留“高 4 位”。
tempC = (float)msb + ((msb >= 0) ? frac : -frac);//把整数和小数部分拼接起来。
return true;
}
//读取温度（摄氏度）

  1. SPI 接口
    SPI 即串行外围设备接口，是一种适用于短距离、设备到设备的高速全双工同步通信接口，通信结构为一主多从，硬件接口上通常由4根信号线组成，简单易用；
    通信模式：
    标准SPI接口一般使用4条线(4个信号)：

NSS/CS：主机片选从机使能信号，通常低电平有效由主机控制；

SCLK：时钟信号，主机产生用于控制数据交换的时机以及速率；

MISO：主机数据输入，从机数据输出，主要用于SPI 设备接收数据；

MOSI：主机数据输出，从机数据输入，主要用于 SPI 设备发送数据。

具体流程：
① 主机拉低目标通信从机片选SS信号线，表示与该设备进行通信；

② 主机产生SCLK时钟信号，告诉从机要交换数据；

③ 主机(Master)把需要发送的数据写入到发送数据缓存区(Menory)后，再打入串行移位寄存器(一般为8位)，串行移位寄存器通过MOSI信号线将字节一位一位地移出传送到从机，同时，MISO接口接收到的数据经过串行移位寄存器一位一位的移到接收缓存区。

从机(Slave)也将其串行移位寄存器中的数据通过MISO信号线发送到主机，同时通过MOSI信号线接收主机发送的数据。

如此一来，两个移位寄存器中的数据就被交换。

  1. CAN 协议
    这是一个多主总线的协议，就想广播那样，一个发射端发出数据，所有的其他接收端都可以接收数据。
    按 标识符长度 划分数据帧，分为标准帧和扩展帧两类：
    标准帧（Standard Frame）：使用 11 位标识符（ID）（共 44～108 位，取决于数据长度）
    [SOF] [11位ID] [RTR] [IDE=0] [r0] [DLC(4位)] [Data(0～8字节)] [CRC] [ACK] [EOF]
    扩展帧（Extended Frame）：使用 29 位标识符（ID）
    [SOF] [11位基础ID] [SRR=1] [IDE=1] [18位扩展ID] [RTR] [r1][r0] [DLC(4位)] [Data(0～8字节)] [CRC] [ACK] [EOF]

  2. [SOF] — Start of Frame（帧起始）
    长度：1 位
    电平：显性（0）
    作用：标志一帧报文的开始，同步所有节点的位时序。
    注意：所有 CAN 节点检测到总线空闲后，若看到 SOF（下降沿），即开始接收该帧。

  3. [11位基础ID] — Base Identifier（基础标识符）
    长度：11 位
    作用：扩展帧 ID 的高 11 位，与标准帧 ID 格式兼容。
    意义：用于仲裁优先级的前 11 位比较；也用于与标准帧共存时的优先级判定。

  4. [SRR] — Substitute Remote Request（替代远程请求位）
    长度：1 位
    固定值：隐性（1）
    作用：
    在扩展帧中代替标准帧 RTR 的位置。
    确保当标准帧与扩展帧使用相同 11 位 ID 前缀时，标准帧优先级更高（因为标准帧 RTR 可为 0，而 SRR=1 是隐性，会被显性 0 覆盖而退出仲裁）。
    仅存在于扩展帧中。

  5. [IDE] — Identifier Extension Bit（标识符扩展位）
    长度：1 位
    值含义：
    0（显性） → 标准帧
    1（隐性） → 扩展帧
    作用：区分标准帧与扩展帧的关键标志位。
    位置：紧跟在 11 位基础 ID 之后。
    在扩展帧中，IDE = 1。

  6. [18位扩展ID] — Extended Identifier（扩展标识符）
    长度：18 位
    作用：扩展帧 ID 的低 18 位，与前面 11 位组合成 29 位完整 ID。
    总 ID 范围：0x00000000 ～ 0x1FFFFFFF（共 2²⁹ 个唯一标识符）
    用途：支持更多设备或更细粒度的消息分类（如汽车中不同 ECU 的子功能）。

  7. [RTR] — Remote Transmission Request（远程传输请求位）
    长度：1 位
    值含义：
    0（显性） → 数据帧（包含 Data 字段）
    1（隐性） → 远程帧（无 Data 字段，请求其他节点发送该 ID 的数据）
    注意：在扩展帧中，RTR 位于扩展 ID 之后。

  8. [r1][r0] — Reserved Bits（保留位）
    长度：2 位（r1 和 r0）
    当前值：必须发送为显性（0）（按 CAN 2.0 规范）
    作用：为未来协议扩展预留。
    接收方行为：若收到非 0 值，可能视为格式错误（取决于控制器实现）。
    🔧 在 CAN FD 中，这些位被重新定义（例如 r0 变为 BRS 位），但在经典 CAN 中必须为 0。

  9. [DLC] — Data Length Code（数据长度代码）
    长度：4 位
    取值范围：0 ～ 15（二进制 0000 ～ 1111）
    实际数据长度映射（经典 CAN）：
    表格
    DLC 值 实际数据字节数
    0～8 = DLC 值

9～15 仍为 8 字节（但某些 CAN FD 控制器可能用 DLC 表示更大长度）
作用：告诉接收方本帧包含多少字节的有效数据。
9. [Data(0～8字节)] — 数据字段
长度：0 到 8 字节（由 DLC 决定）
发送顺序：先发送高字节（MSB），每个字节内先发高位（bit7 → bit0）
内容：应用层要传输的实际信息（如传感器值、控制命令等）
注意：远程帧中此字段不存在。
10. [CRC] — Cyclic Redundancy Check（循环冗余校验）
组成：
15 位 CRC 校验码：基于从 SOF 到 Data 的所有位（不含填充位）计算得出
1 位 CRC 界定符（Delimiter）：固定为 隐性（1）
总长：16 位
作用：检测传输过程中是否发生位错误。接收方重新计算 CRC 并比对，不一致则报错。
11. [ACK] — Acknowledgement（应答段）
组成：
ACK Slot（应答槽）：1 位
发送方发 隐性（1）
任一正确接收的节点必须在此位发 显性（0） 以确认
ACK Delimiter（应答界定符）：1 位，固定为 隐性（1）
总长：2 位
作用：至少一个接收者确认成功接收。若 ACK Slot 仍为 1，发送方认为传输失败。
12. [EOF] — End of Frame（帧结束）
长度：7 位
电平：全部为 隐性（1）
作用：
明确标识本帧结束
提供帧间间隔（IFS）的起点
不允许位填充（即使出现 6 个连续 1 也不插入 0）
之后：进入 IFS（Interframe Space，3 位隐性），然后总线才可被新帧占用。

MCU内存结构（非外设）

ROM 程序存储寄存器

这个内存模块用来存储编译进去的所有对cpu的命令 比如加减操作，另外， 有一个PC（程序计数器，属于CPU的一个特殊寄存器）用来记录当前命令的执行进度 每一条命令执行之后，PC++。

RAM 数据存储寄存器

这个内存模块用来存储 变量 / 寄存器 / 临时数据，cpu可以根据ROM的指令对它进行编辑。
